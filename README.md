# MyManga

This is the expanded **System Architecture Blueprint** for the MangaMood repository. It is designed to be the primary reference for any LLM working on this codebase to ensure accuracy and prevent unnecessary token consumption.
# 📖 MangaMood System Architecture (Phase 3)
MangaMood is a high-performance, API-waterfall-driven manga discovery engine. It uses a custom **NLU-to-SearchPlan pipeline** to bridge user intent with multi-source manga APIs.
## ⚠️ MANDATORY LLM RULES
 1. **Do NOT read properties.js or harvested_knowledge.js:** These are static, massive data files (12k+ lines). They provide the CONCEPT_PROPERTIES and HARVESTED_RULES. They contain **no logic**, only data. If you need to understand how the scoring system works, look at recommendationScorer.js. If you need to know how the pipeline works, look at pipeline.js or searchPlanner.js.
 2. **Bypass Rule:** Always prefer the "Preset" path (triggerPresetSearch) for mood buttons. It avoids the NLU pipeline, saving compute time and avoiding potential parsing errors with niche genres.
 3. **Circuit Breaker Rule:** Never attempt to "fix" a fetch timeout in the API adapters without respecting the session-level mangaDexUnreachable flag in js/mangadex.js.
## 🏗️ Core Architecture Overview
### 1. The Search Lifecycle
 1. **Input:** User text (typed) OR Genre Preset (button).
 2. **Planner:** js/parser/searchPlanner.js transforms the MangaIntent into a flat SearchPlan.
 3. **Waterfall:** js/search.js fetches data from AniList (Tier 1) → Jikan → Kitsu → MangaDex (Tier 4).
 4. **Scorer:** js/parser/recommendationScorer.js ranks results using cosine similarity against stored "mood vectors."
## 📂 File Manifest & API Reference
### js/parser/ (The Brain)
 * **dictionary.js**: Maps synonyms to concept IDs. *ReadOnly Data.*
 * **searchPlanner.js**: Normalizes intent into an API-agnostic SearchPlan.
   * buildPlanFromGenreList(): **The Bypass Path.** Creates a SearchPlan directly from genres.
 * **recommendationScorer.js**: The scoring logic.
   * *Cost Optimization:* Uses Promise.all for parallel Firestore profile fetches. **Crucial:** It performs no network-bound work if the intent lacks a mood profile.
 * **pipeline.js**: The NLU engine. Converts raw text to MangaIntent.
 * **properties.js & harvested_knowledge.js**: **MASSIVE DATA FILES.** Do not read. They contain the CONCEPT_PROPERTIES and HARVESTED_RULES dictionaries.
### js/ (The Engine & UI)
 * **search.js**: The central controller.
   * runSearch(): The atomic execution unit.
   * triggerSearch(): Typed entry point (uses NLU pipeline).
   * triggerPresetSearch(): Button entry point (bypasses NLU).
 * **moods.js**: Manages the 50-mood grid rotation.
 * **mangadex.js**: Tier 4 fallback. Contains session-level mangaDexUnreachable circuit breaker.
 * **main.js**: The bootstrap loader. Binds all search functions to window.
## ⚡ Token & Latency Optimization Guide
| Feature | Cost Control Strategy |
|---|---|
| **API Waterfall** | Tiered sequence ensures expensive/slow APIs (MangaDex/Jikan) only trigger if Tier 1 (AniList) fails. |
| **Circuit Breakers** | Once MangaDex fails twice, it is ignored for the session (prevents repeated timeout costs). |
| **Scoring** | Bypasses Firestore (mangaProfiles.js) if no mood profile vector is present in the SearchPlan. |
| **Cache** | Only Tier 1 (AniList) results are saved to Firebase. Keeps cache database lean. |
## 🛠️ Troubleshooting for LLMs
### 1. The "Genre/Mood" Issue
If a genre button isn't working, it's usually because the NLU parser is trying to match it to a "mood concept" that doesn't exist in the massive data files.
 * **Solution:** Use triggerPresetSearch in js/search.js. It builds a SearchPlan directly from the genres, which is a hardcoded, non-failing operation.
### 2. Modifying the Scorer
 * The WEIGHTS constant in recommendationScorer.js determines how final scores are calculated.
 * If you need to add a new scoring signal, update scoreOne() and the components array in recommendationScorer.js.
 * **Important:** Keep the matchBreakdown keys consistent with js/renderer.js to ensure the "Why?" panel UI does not break.
### 3. Adding New Moods
 * Update the allMoods array in js/moods.js.
 * The query string must be comma-separated genres. They are passed to buildPlanFromGenreList and normalized via GENRE_NORMALIZE in searchPlanner.js. If you add a new genre, ensure it is represented in GENRE_NORMALIZE if the spelling is non-standard.
## 🚀 Deployment Checklist
 1. **Data Files:** harvested_knowledge.js is generated by an external process. If missing, search.js will gracefully degrade to using only properties.js.
 2. **Bindings:** Always verify that new trigger functions are bound to window in main.js.
 3. **Circuit Breaker State:** The mangaDexUnreachable flag resets only on page reload. This is intentional to save token/bandwidth costs during a single user session.


## 💾 Data Handling & Memory Management
This repository uses a specific strategy to handle large datasets. **LLMs should strictly adhere to these rules** to prevent excessive token usage and maintain system performance.
### 1. Handling Large Dictionaries (properties.js & harvested_knowledge.js)
These files are excluded from standard read operations for a reason:
 * **Structure:** They contain JSON-like object structures used for mapping genre concepts to mood weights.
 * **LLM Protocol:** If you are asked to change the scoring logic, **do not read these files**. Instead, assume the existence of an object CONCEPT_DICTIONARY which merges CONCEPT_PROPERTIES and HARVESTED_RULES.
 * **Why:** They represent the "knowledge base." Modifying them is a "harvesting" task (done by a separate Node.js process), not an "application" task. If you need to debug a score, modify the logic in recommendationScorer.js rather than trying to adjust the weights inside these massive data files.
### 2. Cache Strategy
To keep Firestore read/write costs at a minimum:
 * **Tier 1 Only:** We only cache successful AniList results. Lower-tier results (Jikan, Kitsu, MangaDex) are transient and never cached.
 * **Key Generation:** Caches are indexed via generateCacheKey(query, page). If you are debugging cache hits, check firebase.js for the hashing logic.
## 📈 Search Performance & Cost Metrics
To maintain the "token-cost-efficient" goal, consider these metrics when proposing changes:
 1. **Network Overhead:**
   * **The Waterfall:** The search starts at AniList. It only progresses to Jikan if the AniList array is empty. This prevents redundant API requests.
   * **Async Scoring:** The scoreResults function uses Promise.all to fetch profiles. This is a "fan-out" operation. **Do not convert this to a sequential loop**, or you will increase the latency (and potential timeout risks) by a factor of the result count (usually 10-20).
 2. **Circuit Breaker Logic (mangadex.js):**
   * The mangaDexUnreachable flag is a **"Nuclear Option."** Once toggled true, it stays true for the life of the tab. This saves 3 seconds of AbortController timeout waiting per card.
   * **LLM Caution:** Do not add logic that resets this flag inside a promise chain unless it is triggered by an explicit user "Reconnect" action.
## 🔌 Integration Summary for UI/UX Developers
If you are modifying the UI, these are the hooks you must use to keep the search engine responsive:
 * **For "Why is this ranked here?":** Use the matchReasons array returned by scoreResults. It is already pre-deduplicated.
 * **For "Loading" states:** Always invoke renderSkeletonLoaders(15) before calling any triggerSearch variant. This provides immediate visual feedback and avoids "empty grid" jank.
 * **For "Mood" buttons:** Always link buttons to triggerPresetSearch in moods.js. This is the only way to ensure the search bypasses the NLU pipeline and lands directly in the SearchPlan builder.
## 🔮 Future-Proofing (Maintenance)
 * **When adding a new API Tier:** Add a new helper in search.js (e.g., fetchFrom[NewSource]), ensure it consumes the SearchPlan object, and add the Tier check in the runSearch waterfall.
 * **When adding a new scoring weight:** You **must** update the WEIGHTS constant in recommendationScorer.js to ensure the total sum of weights remains balanced, and add the weight to the components filter array.
